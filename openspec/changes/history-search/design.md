## Context

PaseBoard 当前有历史记录面板，可展示最近 1000 条粘贴板历史。用户需要快速查找特定内容，目前只能逐条滚动浏览。需要实现类似 vim 的即时搜索模式，让用户输入关键词快速过滤。

当前状态：
- 历史记录面板已实现（HTML/CSS/JS）
- 前端使用原生 JavaScript（无框架）
- 已有键盘事件处理（如 ESC 关闭面板）

## Goals / Non-Goals

**Goals:**
- 用户按 `/` 键快速进入搜索模式
- 输入字符即时过滤历史记录，无需回车确认
- ESC 键退出搜索模式，恢复正常浏览
- 搜索过程中用户输入不中断（异步过滤）
- 搜索模式中 `/` 作为普通字符输入

**Non-Goals:**
- 不实现复杂的搜索语法（正则、布尔逻辑）
- 不支持搜索历史记录的元数据（设备名、时间等）
- 不优化大列表渲染性能（当前 1000 条上限足够）
- 不支持多关键词组合搜索

## Decisions

### 决策 1：搜索模式状态管理

**选择：** 使用单一状态变量管理搜索模式

**方案对比：**
- **方案 A（选择）：** 单一 `searchMode` 布尔值 + `searchQuery` 字符串
  - 优点：简单、易调试、状态清晰
  - 缺点：状态耦合
- **方案 B：** 使用有限状态机（FSM）
  - 优点：状态转换明确
  - 缺点：过度设计，当前场景只需两种状态（normal/search）

**理由：** 当前只有两种模式，不需要 FSM 的复杂性。

### 决策 2：过滤实现策略

**选择：** 前端纯 JavaScript 过滤，使用 `requestAnimationFrame` 优化

**方案对比：**
- **方案 A（选择）：** 输入时直接过滤，使用 rAF 确保渲染不阻塞
  - 优点：即时反馈、实现简单
  - 缺点：超长列表可能卡顿（但 1000 条上限足够）
- **方案 B：** 防抖（debounce）后过滤
  - 优点：减少计算频率
  - 缺点：用户感知延迟，体验不佳
- **方案 C：** Web Worker 后台过滤
  - 优点：不阻塞 UI 线程
  - 缺点：复杂度高，1000 条不需要

**理由：** 1000 条记录的字符串匹配在现代浏览器中 < 1ms，直接过滤足够。

### 决策 3：搜索匹配算法

**选择：** 子字符串匹配（`String.includes()`）

**方案对比：**
- **方案 A（选择）：** `item.content.toLowerCase().includes(query.toLowerCase())`
  - 优点：简单、直观
  - 缺点：不支持模糊匹配
- **方案 B：** 模糊匹配库（如 Fuse.js）
  - 优点：容错性好
  - 缺点：外部依赖、性能开销

**理由：** 粘贴板内容是精确复制，用户搜索时通常知道确切关键词。

### 决策 4：键盘事件处理

**选择：** 在全局 keydown 事件中拦截 `/` 键，搜索模式下不拦截

**方案对比：**
- **方案 A（选择）：** 条件判断 `if (!searchMode && key === '/')`
  - 优点：集中管理、逻辑清晰
  - 缺点：需要维护状态
- **方案 B：** 搜索模式下移除全局事件，绑定搜索框事件
  - 优点：事件解耦
  - 缺点：事件绑定/解绑开销、可能丢失事件

**理由：** 条件判断更简单，避免动态事件绑定的复杂性。

## Risks / Trade-offs

**风险 1：搜索输入框失去焦点**
- 风险：用户按其他快捷键时，搜索框可能失去焦点
- 缓解：搜索模式下，所有非 ESC 按键都重新聚焦搜索框

**风险 2：大量匹配结果时的渲染性能**
- 风险：如果查询过于宽泛（如单个字母），可能匹配数百条
- 缓解：可选地限制显示结果数（如前 100 条），但当前 1000 条上限足够

**风险 3：移动端支持**
- 风险：移动端没有物理键盘，无法触发 `/` 键
- 缓解：可增加搜索图标按钮作为备用入口（但不在本次范围内）
