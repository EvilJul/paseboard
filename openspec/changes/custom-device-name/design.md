## Context

PaseBoard 当前自动使用系统主机名作为设备名称（如 "fushengdeMacBook-Pro"）。用户需要自定义设备名称以便于识别。当前配置文件已存在，需要扩展以支持自定义名称。

当前状态：
- 设备名称从系统主机名自动获取
- 配置文件存储设备 ID、端口等信息
- 前端设备列表显示设备名称
- 历史记录显示来源设备名称

## Goals / Non-Goals

**Goals:**
- 用户可自定义本机设备名称
- 名称修改后立即生效（无需重启应用）
- 名称持久化存储（重启后保留）
- 只允许修改自己的设备名称
- 输入验证：长度限制（1-30字符）、禁止特殊字符

**Non-Goals:**
- 不支持修改其他设备的名称
- 不支持设备名称的唯一性校验（允许多设备同名）
- 不支持设备头像或图标自定义
- 不支持名称历史记录或回滚

## Decisions

### 决策 1：设备名称存储位置

**选择：** 扩展现有配置文件（~/.paseboard/config.json）

**方案对比：**
- **方案 A（选择）：** 在现有 config.json 中添加 `custom_device_name` 字段
  - 优点：复用现有配置管理逻辑、简单
  - 缺点：配置文件结构变化
- **方案 B：** 使用独立的配置文件（~/.paseboard/device_name.txt）
  - 优点：隔离性好
  - 缺点：增加文件管理复杂度
- **方案 C：** 存储在 SQLite 数据库中
  - 优点：统一数据管理
  - 缺点：过度设计，配置信息不适合存数据库

**理由：** 配置文件已有成熟的读写逻辑，扩展最简单。

### 决策 2：名称生效时机

**选择：** 修改后立即生效，通过内存状态同步

**方案对比：**
- **方案 A（选择）：** 修改配置文件 + 更新内存中的设备名称 + 触发事件通知前端
  - 优点：即时反馈、用户体验好
  - 缺点：需要状态同步机制
- **方案 B：** 修改配置文件 + 重启应用生效
  - 优点：实现简单
  - 缺点：用户体验差，需要重启

**理由：** 即时生效是基本用户体验要求。

### 决策 3：前端 UI 入口

**选择：** 在设备列表的"本机"设备行添加编辑按钮

**方案对比：**
- **方案 A（选择）：** 设备列表中本机设备行显示编辑图标，点击弹出输入框
  - 优点：直观、符合用户心智模型
  - 缺点：需要识别本机设备
- **方案 B：** 在设置弹窗中添加设备名称修改
  - 优点：集中管理
  - 缺点：不够直观，用户可能找不到
- **方案 C：** 双击设备名称直接编辑
  - 优点：快捷
  - 缺点：可能误触发，不明显

**理由：** 在设备列表中直接编辑最直观，用户能看到修改效果。

### 决策 4：输入验证规则

**选择：** 1-30字符，允许中文、英文、数字、空格、连字符

**方案对比：**
- **方案 A（选择）：** 简单的字符白名单验证
  - 优点：简单、安全
  - 缺点：可能限制某些特殊字符
- **方案 B：** 仅限制长度，不限制字符
  - 优点：灵活
  - 缺点：可能引入安全问题（如 XSS）

**理由：** 限制字符可避免显示问题和安全风险。

## Risks / Trade-offs

**风险 1：设备名称冲突**
- 风险：多设备使用相同名称，用户难以区分
- 缓解：设备列表仍显示 IP 地址作为辅助信息，未来可考虑添加设备 ID 后缀

**风险 2：名称长度过长导致 UI 显示问题**
- 风险：长名称可能破坏设备列表布局
- 缓解：限制最大 30 字符，前端使用 CSS 截断（text-overflow: ellipsis）

**风险 3：配置文件损坏**
- 风险：写入配置文件时可能损坏（如磁盘满、进程被杀）
- 缓解：写入前备份、使用原子写入（先写临时文件再 rename）

**风险 4：网络广播延迟**
- 风险：修改名称后，其他设备可能需要几秒钟才能看到新名称
- 缓解：mDNS 广播间隔 5 秒，最多延迟 5 秒，可接受
